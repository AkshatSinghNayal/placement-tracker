# OfferForge Backend Migration — FastAPI/Postgres → Express/MongoDB (JavaScript)

## 1. Migration strategy (overview)

Replace the entire `backend/` Python stack (FastAPI, SQLAlchemy, Alembic, Pydantic, Postgres, asyncpg) with a **pure JavaScript (ES Modules)** backend: **Node 20 + Express + MongoDB 7 + Mongoose**, **JWT** auth, **passport-google-oauth20** for Google OAuth, **multer + Cloudinary** for resume PDFs.

The React frontend (`frontend/`) stays **byte-for-byte unchanged**. Success = `docker compose up --build` brings up React @ 5173, Express @ 8000, Mongo @ 27017, and every existing screen works (Dashboard, Companies, DSA, Notes, Resources, Resumes, Search ⌘K, Analytics, Google login, password reset).

**Guiding principles**
- Preserve the exact API contract: `/api/v1` prefix, HTTP methods/paths, JSON response shapes, pagination shape `{items,total,limit,offset}`, enum string values, `Authorization: Bearer <jwt>`, error shape `{detail}`.
- Reuse the existing **seed company data** (migration `0002_seed_companies.py`) by porting it to a JS seeder.
- Map every SQLAlchemy model to a Mongoose model using native **ObjectId** refs, `timestamps: true`, and indexes mirroring the Postgres indexes/constraints.

## 3. Critical fidelity notes (these break the UI if missed)

1. **Refresh-cookie auth is mandatory.** The frontend stores the access JWT **only in memory**; on every page reload `AuthContext` calls `POST /auth/refresh` (httpOnly cookie), and a 401 interceptor silently refreshes. So `/auth/refresh` **must return the full `{access_token, token_type:"bearer", user}`** and read an httpOnly `refresh_token` cookie. → I keep opaque refresh tokens (hashed, in a `RefreshToken` collection, rotated on each refresh). *Deviation from your `.env` (which lists only `JWT_EXPIRES_IN=7d`):* I'll add `ACCESS_TOKEN_TTL_MINUTES=15` + `REFRESH_TOKEN_TTL_DAYS=7` and set `JWT_EXPIRES_IN=15m`, because a 7-day access token makes the refresh flow pointless; documented in `.env.example`. `withCredentials:true` on the frontend → CORS `credentials:true`, origin = `FRONTEND_URL`, cookie `httpOnly; SameSite=Lax` (Lax in dev so the OAuth top-level redirect carries it; switchable to `None;Secure` for cross-origin prod).
2. **`/api/v1` prefix** on every route; Express mounted at `/api/v1`.
3. **Google OAuth redirect targets** (passport, session-less): on success set refresh cookie + `302 → ${FRONTEND_URL}/auth/google/callback?success=1`; on failure `302 → ${FRONTEND_URL}/login?error=oauth_declined|oauth_failed|oauth_state_mismatch|oauth_missing_params`.
4. **Resume upload**: multipart, field `file`, field `version_label`, PDF-only, ≤5MB → Cloudinary → store `cloudinary_url` + `cloudinary_public_id` in the `Resume` doc. `GET /resumes/{id}/pdf` streams the PDF (redirect to Cloudinary raw URL with `Content-Disposition: inline`).
5. **Exact response shapes** preserved (computed fields stay read-time): resume `readiness_score = coverage*0.6 + (active?40:0)`; dashboard `overall_progress = dsa*0.5 + resume*0.3 + checklist*0.2`; `checklist_progress_pct`; `iso_week` format `YYYY-Www`.
6. **Enums** must match string-for-string (`'Not Started'`, `'Marked for Revision'`, `'Product-based'`, etc.) — centralized in a `src/constants/` module shared by models + validators.
7. **Soft constraints** enforced in app code (no native partial-unique index equivalent): exactly one active resume per user; case-insensitive unique tag name; note attach XOR (company xor dsa_problem).
